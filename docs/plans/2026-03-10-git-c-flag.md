# Use git -C Instead of cd Chains in Lightbulb Skill -- Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Replace `cd <path> && git <command>` patterns in the lightbulb skill with `git -C <path> <command>` to avoid triggering Claude Code's bare repository security prompts when running git operations in worktrees.

**Architecture:** All changes are to a single file: `skills/lightbulb/SKILL.md`. The changes:
1. Introduce a `WORKTREE_PATH` variable convention so all subsequent git commands can reference it
2. Rewrite every orchestrator git command (steps 2, 3, 5, 6) to use `git -C "$WORKTREE_PATH"` instead of assuming the shell is `cd`'d into the worktree
3. Add a note about subagent context (subagents dispatched via Agent tool already run in the worktree directory, so they are unaffected)

**Tech Stack:** Claude Code skill (Markdown), no code dependencies.

**Out of scope:** The `superpowers:using-git-worktrees` skill is an external dependency (from the `superpowers` plugin) and cannot be modified by this project.

---

## Background

Claude Code's sandbox flags compound `cd <path> && git <command>` as potentially dangerous to prevent bare repository attacks. This triggers approval prompts even when the target is a worktree *inside* the current trusted repository -- a false positive.

The lightbulb skill orchestrator runs git commands (add, commit, push, diff, rev-parse, symbolic-ref) after the worktree is created by `superpowers:using-git-worktrees`. Since bash state does not persist between tool calls in Claude Code, each git command that targets the worktree currently requires a `cd "$WORKTREE_PATH" && git ...` prefix, and each such compound command triggers the security prompt.

The `git -C <path>` flag runs git against a different directory without changing the shell's working directory, which avoids triggering the bare repository protection entirely.

**What changes and what doesn't:**
- **Orchestrator git commands** (in SKILL.md code blocks): Must change from assuming `cd` context to using `git -C "$WORKTREE_PATH"`.
- **Subagent commands**: Subagents are dispatched via the Agent tool and run in the worktree context managed by the Agent tool's working directory. Their git commands do not need `git -C` since they are already "inside" the worktree. No changes needed to subagent prompt templates.
- **Non-git commands** (gh, npm, etc.): These don't trigger the security prompt, so they are out of scope.

## Task 3: Update Step 5 push to use git -C

**Files:**
- Modify: `skills/lightbulb/SKILL.md`

### Step 1: Replace the git push command in Step 5

In `skills/lightbulb/SKILL.md`, find the push and PR creation block in Step 5. It currently reads:

```
git push -u origin <branch-name>
gh pr create --draft --title "<issue-title>" --body "$(cat <<'EOF'
```

Replace the git push line with:

```
git -C "$WORKTREE_PATH" push -u origin <branch-name>
gh pr create --draft --head <branch-name> --title "<issue-title>" --body "$(cat <<'EOF'
```

Note: `gh pr create` needs `--head <branch-name>` because it infers the head branch from the CWD's git state. Since the orchestrator's CWD is the main repo (not the worktree), `gh` would default to `main` without the explicit `--head` flag. Other `gh pr` commands (`comment`, `checks`, `ready`, `merge`) take explicit PR numbers and are CWD-independent.

### Step 2: Verify

Run: `grep -n "git.*push.*origin" skills/lightbulb/SKILL.md`

Expected: The line should contain `git -C "$WORKTREE_PATH" push -u origin`.
